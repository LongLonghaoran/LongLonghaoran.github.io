笔记本：CI/CD
什么是gitlab ci/cd呢?
    gitlab ci/cd 是gitlab内建的用于持续开发方法的工具:
        continuous integration 持续集成
        持续交付、分发
        持续部署

持续集成：每一个推送代码都会运行一次管线脚本来构建、测试、验证代码，在合并之前
持续交付和部署由更进一步的ci组成，即mei次推送到默认分支的时候都部署到生产环境

好处：更早的发现bug和错误

通过自动化运维gitlab还可以自动探测、构建、测试、部署并监控的你的应用

一些概念：
  •     管线： 结构化CI/CD流程
  •     CI/CD变量： 复用键值对
  •     环境：将应用部署到不同环境中（比如演示环境、生产环境）
  •     任务工件：输出、使用、复用任务工件
  •     缓存的依赖：将依赖缓存用于更快的执行
  •     Gitlab执行器：配置你自己的执行器来执行你的脚本
  •     管线效率：配置你自己的管线来快速有效的运行
  •     测试用例：配置自己的关系来快速有效的运行

执行器讲解：
    要执行管线就需要配置执行器来执行,它就是运行管线中的任务的程序。
    要安装执行器的话建议将其安装在独立与gitlab实例的机器上，这样利用性能和安全。当你分离了机器之后，你就可以有不同的操作系统和工具了，比如k8s和docker
    安装说明：https://docs.gitlab.com/runner/install/index.html
    安装好执行器后需要注册才可以使用
    执行器是用go语言开发的。
    执行者executor:
        所谓执行决定了每个任务运行的环境，比如：
            如果希望用powershell来运行命令，那么你或许可以在windows server上runner并且注册执行器让它使用shell执行者
            如果希望用docker容器，你或许会在linux服务器上安装并注册它来使用docker执行者
    标签：
        注册好执行器后，可以给它添加标签，用于确定任务与可执行该任务的执行器之间的关系
    配置执行器:
        配置文件位于config.toml，该文件在执行器安装过程中就安装好了
    监控执行器：可以使用promethus来配合监控
    使用执行器来运行任务：
        执行器安装并配置好之后，在.gitlab-ci.yml文件中指定好执行器的名称或者标签名。然后，当你提交到仓库的时候，管线开始运行，接着执行器的执行者开始处理命令
    执行器的一些特性： 
            多任务并行、
            多服务器多令牌结构（甚至每个项目一个令牌），令牌的概念下面会解释、对每个令牌限制并行数量、
            任务可以：本地、docker容器、通过ssh使用docker容器、连接远程服务器ssh来执行
            支持bash、powershell
            GNU/linux、macOS、windows（其实几乎任何能够运行docker的主机都可以,因为任务可以放在docker中执行）
            可以作为后台服务来运行
            支持通过http服务来使用普罗米修斯监控
        执行器工作流程：
            包含了注册、认证以及任务令牌的操作
            注册令牌：即注册时gitlab 设置页面中的注册令牌
            认证令牌：通过gitlab api注册成功后返回的令牌，或者通过gitlab-runner命令注册后，该令牌也会自动添加到配置文件中去
            任务令牌：
            访问令牌：即调用gitlab api的时候需要传递的令牌，这个也可以通过在Settings -> Access Tokens中进行设置

在gitlab中使用执行器：
    在gitlab cicd中执行器会执行在.gitlab-ci.yml中的定义的代码。执行器是一个轻量级的、高度可扩展的代理，通过gitlab cicd的api来拾取任务任务并执行，并将结果发送回gitlab 实例。
    有3种执行器： 公共执行器、组执行器、特定执行器
    公共执行器只有超级管理员可以更改
    组执行器用于同一个组中的所有项目可以访问同一组执行器，组执行器队列遵循先进先出原则
    组执行器创建步骤：Settings -> CI/CD -> 展开Runners部分 -> 注意URL和token -> 注册执行器
    管理、查看组执行器：
            可通过runner api来管理
    特定的执行器:
        特定于项目的执行器，注意fork的项目不会自动共享特定的执行器
        创建： 安装执行器-》Settings -> CI/CD -〉 展开Runners部分 -》 注意URL和Token -〉注册执行器
    给特定的项目启用特定的执行器
        必须有该项目的所有者全县
        特定的执行器不能是锁定的
    通过标签来限制的任务执行的数量，默认情况下执行器只会拾取打了标签的任务，你必须拥有所有者全县才可以更改，
    如果要让执行器拾取未打标签的任务：
        Settings>CI/CD 展开Runners部分 -> 找到你要更改的执行器并确保它是一启动的-》点击铅笔按钮-〉检查 Run untagged jobs选项-》点击来更改
        注意： 执行标签列表不能为空，当它不允许去拾取未打标签的任务时
    通过变量来配置执行器的行为：
        在.gitlab-ci.yml文件通过variables:部分中设置一些变量可以配置执行器的行为：
  • GIT_STRATEGY git 拉取方式
  • GIT_SUBMODULE_STRATEGY 子模块拉取方式
  • GIT_CHECKOUT
  • GIT_CLEAN_FLAGS
  • GIT_FETCH_EXTRA_FLAGS
  • GIT_DEPTH (shallow cloning) 克隆拉取时的深度
  • GIT_CLONE_PATH (custom build directories) 自定义克隆到的目录



执行器最佳实践：
    取决于具体的执行者，其构建目录会有所不同：
    k8s docker docker machine: 在/builds
    shell在 $PWD/builds
    ssh virtualbox parallels:在~/builds,home目录取决于自行配置的用于ssh连接所使用的home目录
    custom:必须通过builds_dir设置明确指定构建目录
    关闭执行器:
    执行器会单独启动一个进程来执行注入下载、上传工件、处理缓存的事情。这些进程使用gitlab-runner命令来执行，可以通过pkill -QUIT gitlab-runner 或者killall QUIT gitlab-runner来杀掉它（进程的执行命令会有gitlab-runner，pkill和killall都可以通过名称匹配来杀进程)，但是这样杀掉之后，所处理的操作也会执行失败，所以有2种方式来避免：
    1.将执行器注册为本地服务systemd，使用SIGQUIT作为杀进程的信号，然后使用gitlab-runner stop 或者 systemctl stop gitlab-runner.service来关闭执行器
    2.手动执行发送信号
        kill -SIGQUIT pid (pid在gitlab-runner run的时候会打印到标准输出（终端）)
