移动web!加油
之前已经接触过h5 c3的应用，它的主要应用方向是移动端开发,其实光说写样式的话，
已经够了，学习移动端的主要需要解决的问题还是移动端的适配问题
移动端布局的一些特点:
1.基本不用固定像素，而是使用百分比
我们将通过几个例子来学习移动端适配:
    1.京东在线商城
        1个通栏的轮播图
        2排首页icon
        采用"流式布局",即采用百分比
        同时对移动端的viewport视口进行设置
流式布局:
    流式布局就是百分比布局，即： 非固定像素，内容向两侧填充，可以理解成流动的布局，称为流失布局
viewport 视口:
    对移动端的viewport进行设置就能达到适配的目的，这是一个虚拟的区域，用于承载网页的。
    以往在Pc端中，我们可以说整个网页是承载在浏览器窗口中的，浏览器的宽度决定了包含网页的容器的宽度
    而在移动端中，是viewport承载了网页，而浏览器承载了viewport
适配方案:
    1.网页内容的宽度必须和浏览器保持一致
    2.默认显示的缩放比例和pc端保持一致(缩放比例1.0)
    3.不允许用户自行缩放网页
    满足这些要求就达到了适配，国际上通用的适配方案
    适配设置:
        <meta name='viewport'>设置视口的标签，写在head里面并且应该紧接着编码设置
        即:
            <meta charset="UTF-8">
            <meta name="viewport">
        viewport的功能:
        1.可以设置宽度、高度、默认的缩放比例、是否允许用户自行缩放、最大缩放比例、最小缩放比例
        2.设置方法
            width 宽度 heigth 高度, 注意： 设置时不需要带单位
                width=320 将宽度设置为320，但是不同手机宽度不一致，所以这样仍然不会是适配
                可以设置宽度为与设备宽度一致:
                width=device-width
                还需注意一点: 在移动端设备上，device-width的宽度并非手机屏幕的分辨率像素宽度，而是视口展示的宽度
            initial-scale 初始的/默认的缩放比例,这个缩放比例指的是相对于视口大小的缩放比例
            user-scalable 用户是否可以自行缩放, 注意：取值是 yes / no，而不是 true/false
            maximum-scale 最大缩放比例
            minimum-scale 最小缩放比例
            通过content属性来设置
            一个标准的适配的示例:
                可以通过通过快捷方式生成 meta:vp + tab
                <meta name="viewport"
                      content="width=device-width, user-scalable=no, initial-scale=1.0, maximum-scale=1.0, minimum-scale=1.0">
                <meta name="viewport" content="width=device-width initial-scale=1.0 user-scalable=no">
                                                视口宽度=设备宽度 缩放比例=1        不允许用户缩放
非主流的适配方案:
    关于屏幕像素比:
        什么是像素(物理像素) px(页面的尺寸单位)
        物理像素 是设备显示屏的最小可视颗粒的大小，也就是人肉眼可以看到的最小的像素点
        而像素比指的就是 1px中可以放入的物理像素点的数量
    在高像素比的屏幕下，非矢量的图片显示显示效果会变模糊
    所以为了在高像素比下也能显示较好，有的网页会将页面画的比屏幕尺寸大很多，然后
    再根据屏幕尺寸进行缩小缩放，比如屏幕像素比如果是2，那么就缩小2倍，将initial-scale设为0.5，像素比是3，就缩小3倍，将initial-scale设为0.3333
    但是，这样的适配方案成本非常高
不建议在移动端使用Jquery
    1.移动端主流浏览器
        谷歌 safari 欧朋...
        这些浏览器都有一个特点，内核基本上都是webkit 或者 blink
    2.而Jquery做了大量针对IE的兼容,但是在移动端这不需要
    3.移动端使用 一个叫做 zepto.js的库,它也解决了很多兼容问题
由于移动端采用的流失布局，而且屏幕尺寸都不一致，所以需要设置 box-sizing
    而且一般会这么设置:
        *,
        *::before,
        *::after {
            padding: 0;
            margin: 0;
            box-sizing: border
        }
-webkit-tap-highlight-color: #ccc;
默认情况下载移动端，单击超链接时会闪烁一下颜色,该属性是移动端特有的，可以用于设置高亮瞬间的颜色，也可以设置为transparent透明从而隐藏这种效果
-webkit-appearance: none; 元素的外观
    none: 没有任何样式(内外阴影，立体感）
二倍图:
    使用精灵图时,解决移动端图片失真问题，使用2倍于页面尺寸的图片，放入页面时，使用background-size: 一半的图片宽高 压缩背景图片
轮播图:
    以往为了实现 “无缝滚动”我们会需要n+1张图片，即增加一张放在尾处即可
    但是移动端中无法确定滚动方向，所以需要n+2张图片，前后各塞一张多出来的图片才行
    核心: 需要做到 轮播图的图片可以随着屏幕尺寸的缩放而一起缩放
    原理: 比如有10张图片，那么设置ul的宽度为1000%,然后设置其中的图片宽度为10%
删除线样式:

屏幕的触摸事件
    touch是移动端的触摸事件，而且是一组事件
    touchstart 当手指触摸到屏幕时就触发
    touchmove 当手指在屏幕中滑动时触发
    touchend  当手指离开屏幕时触发
    touchcancel 当被迫终止滑动的时候触发(比如由于其他应用的弹窗，比如来电话了？导致页面触摸事件被终止)
    利用touch相关事件就可以实现移动端常见滑动效果和移动端常见的手势事件
    绑定事件，我们只需要使用addEventListener就可以了，因为我们只需要考虑谷歌浏览器的情况就够了
    js会将触摸的事件对象作为参数传递事件处理函数
        addEventListener(dom, function(event){})
        其中event就是触摸事件对象，我们需要关注其中的3个属性:
        1.changeTouches    类型 TouchList,可以称它为触摸点的集合(一个手指触摸就是一个触发点，2个手指就是2个触发点）
            改变过的触摸点集合，所谓改变过指的就是，比如 手指初次触摸到屏幕，就会有触摸点从无到有
        2.targetTouches    类型 TouchList,可以称它为触摸点的集合
            当前元素的触摸点集合
        3.touches          类型 TouchList,可以称它为触摸点的集合
            页面上所有的触摸点集合，在touchend事件中，由于手指已经离开屏幕，所以targetTouches和touches
            集合都将为空，只有changedTouches不为空
    触摸点:
        需要关注3组属性
        clientX clientY(基于浏览器窗口，而且由于我们已经将视口调整为和页面大小一致，所以也可以理解为基于视口）
        pageX pageY （基于页面，而由于页面和浏览器窗口大小是一致，所以也可以说是基于视口 基于窗口）
        screenX screenY 基于整个屏幕，这个就和前两者不同了，基于屏幕意味浏览器窗口以外的距离也会算进来

轮播图的滑动功能
    原理：通过监听手指滑动屏幕的事件，来同时改动图片的偏移值（比如left rigth，但是我们这里采用的是transform属性的translateX)
