1. 介绍

GPUDirect RDMA技术解决的是计算集群节点间GPU卡跨PCIe总线的通信问题。GPUDirect RDMA是在GPU加速领域的又一个技术深入，结合infiniband技术的原生支持，多个GPU协处理器间的带宽和延迟性能会得到大幅度的提升，从而使CUDA编程环境下的应用加速有了更大的发挥空间。GPUDirect RDMA 技术支持 Server1 的 GPU 内存可以通过同一 root complex 下的网卡直接访问对端 Server2 的 GPU 内存。GPU 通信通道中无需 CPU 参与，大幅度降低了 GPU 间通信的延迟。

[图片/附件]


没有这项技术之前，GPU需要先将数据从GPU内存搬移到系统内存，然后再利用RDMA传输到计算机2，计算机2的GPU还要做一次数据从系统内存到GPU内存的搬移动作。GPUDirect RDMA技术使得进一步减少了GPU通信的数据复制次数，通信延迟进一步降低。所谓GPUDirect RDMA，就是计算机1的GPU可以直接访问计算机2的GPU内存。

3. 使用



3.1. 限制条件

  • GPUDirect RDMA在Tesla和Quadro GPU上都可用。GPU和第三方设备必须共享相同的上游PCI Express root complex。第三方设备可以是：网络接口、视频采集设备、存储适配器。
  • 设备驱动程序需要添加GPUDirect RDMA支持，必须修改内核驱动程序中的地址映射有关的代码（get_user_pages()调用附近）。所以需要英伟达的内核驱动。



4. 核心原理



4.1. GPUDirect RDMA工作原理

在两个peer之间设置GPUDirect RDMA通信时，从PCI Express设备的角度来看，所有物理地址都相同。在这个物理地址空间内，有称为PCI BARs的线性窗口。每个设备最多有六个BAR寄存器，因此可以有多达六个活动的32位BAR区域。64位BAR占用两个BAR寄存器。PCI Express设备读写peer设备的BAR地址的方式与读写系统内存的方式相同。

传统上，像BAR窗口这样的资源被映射到用户或内核地址空间，使用CPU的MMU作为内存映射I/O（MMIO）地址。然而，由于当前操作系统在驱动程序之间交换MMIO区域方面没有足够的机制，NVIDIA内核驱动程序来执行必要的地址转换和映射。



4.2. 标准DMA传输

标准DMA传输，存在以下组件：

  1. 用户空间程序
  2. 用户空间通信库
  3. 用于执行DMA传输的设备的内核驱动程序

一般的顺序如下：

  1. 用户空间程序通过用户空间通信库请求传输。此操作接受数据的指针（虚拟地址）和字节数。
  2. 通信库必须确保与虚拟地址和大小相对应的内存区域已准备好进行传输。如果没准备好，必须由内核驱动程序处理（下一步）。
  3. 内核驱动程序从用户空间通信库接收虚拟地址和大小。然后，它要求内核将虚拟地址范围转换为物理页的列表，并确保它们已准备好进行传输。此操作称为固定内存。
  4. 内核驱动程序使用页列表来配置物理设备的DMA引擎。
  5. 通信库启动传输。
  6. 传输完成后，通信库最终应清理用于固定内存的所有资源。操作称为取消固定内存。


4.3. GPUDirect RDMA传输

要支持GPUDirect RDMA传输，必须对上述序列进行一些更改。首先，引入了两个新组件：

  1. 用户空间CUDA库
  2. NVIDIA内核驱动程序

使用CUDA库的程序需要把地址空间分割为GPU虚拟地址和CPU虚拟地址，通信库必须为它们实现两条分开的路径。

用户空间CUDA库提供了一个函数，允许通信库区分CPU和GPU地址。此外，对于GPU地址，它返回额外的元数据，用于标识由地址表示的GPU内存。有关详细信息，请参阅用户空间API。

CPU地址和GPU地址的路径之间的区别在于内存的固定和取消固定的方式。对于CPU内存，这由内建的Linux内核函数处理（get_user_pages()和put_page()）。但是，在GPU内存的情况下，固定和取消固定必须由NVIDIA内核驱动程序提供的函数处理。有关详细信息，请参阅固定GPU内存和取消固定GPU内存。



4.4. UVA CUDA内存管理基础

统一虚拟寻址（UVA）是一种在CUDA 4.0及以后版本中默认启用的内存地址管理系统，适用于在运行64位进程的Fermi和Kepler GPU上。UVA内存管理的设计为GPUDirect RDMA的操作提供了基础。在支持UVA的配置中，当CUDA运行时初始化时，应用程序的虚拟地址（VA）范围被分成两个区域：CUDA管理的VA范围和操作系统管理的VA范围。所有CUDA管理的指针都在此VA范围内，而该范围始终位于进程VA空间的前40位内。

在CUDA VA空间内，地址可以细分为三种类型：

  • GPU：由GPU内存支持的页面。这将无法从主机访问，相应的VA在主机上永远不会有物理支持。从CPU解引用指向GPU VA的指针将触发段错误。
  • CPU ：由CPU内存支持的页面。这将在主机和GPU上同时可访问，VA相同。
  • FREE ：这些VA被CUDA保留供将来分配使用。

此分区允许CUDA运行时通过指针值在保留的CUDA VA空间内确定内存对象的物理位置。

这些地址在页面粒度上被细分为这些类别；页面内的所有内存都是相同类型的。GPU页面的大小可能与CPU页面的大小不同。CPU页面通常为4KB，而Kepler类GPU上的GPU页面为64KB。GPUDirect RDMA专门在CUDA VA空间内的GPU页面上运行（由cudaMalloc()创建）。

4.6. 内核空间API

数据结构：

typedef
struct nvidia_p2p_page {
    uint64_t physical_address;
    union nvidia_p2p_request_registers {
        struct {
            uint32_t wreqmb_h;
            uint32_t rreqmb_h;
            uint32_t rreqmb_0;
            uint32_t reserved[3];
        } fermi;
    } registers;
} nvidia_p2p_page_t;


typedef
struct nvidia_p2p_page_table {
    uint32_t version;
    uint32_t page_size;
    struct nvidia_p2p_page **pages;
    uint32_t entries;
    uint8_t *gpu_uuid;
} nvidia_p2p_page_table_t;


typedef
struct nvidia_p2p_dma_mapping {
    uint32_t  version;
    enum nvidia_p2p_page_size_type page_size_type;
    uint32_t  entries;

} nvidia_p2p_dma_mapping_t;
