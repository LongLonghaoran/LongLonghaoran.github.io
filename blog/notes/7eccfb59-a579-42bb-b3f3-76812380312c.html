什么是openapi:
    OpenAPI是描述新 API 的最广泛采用的行业标准
    api描述文件,一个API描述文件（有时称为合同）是一种机器可读的API的说明书中。它应该努力做到尽可能完整和详细，尽管绝对的完整性通常不是必需的。此外，就像法律合同一样，它越明确，就越有用，它是用来描述api的
    OpenAPI 规范 ( OAS ) 是基于 HTTP 的远程 API的供应商中立描述格式。它最初基于 SmartBear Software 于 2015年捐赠的 Swagger 2.0 规范。
    可能是因为使用 OpenAPI 时可用的工具数量众多，它已成为描述现代 API 的最广泛采用的行业标准。
    总而言之，它是一套用以定义api的规范

最佳实践：
    1.设计优先（不同于以往先进行开发，然后通过代码注释在生成文档，这种方式不需学习新的编程语言，所以通常被认为是最简单的），设计优先是先编写api描述再编写代码
    2.保持单一真相来源
    3.将openapi文档添加到源码管理
    4.是openapi文档可供用户使用

openapi规范:
[图片/附件]

    基本结构:
        
    api端点:
        是paths字段中很重要的部分，描述了具体的api操作
        其中，字段名是路径，字段值是具体路径项对象

    路径项对象:
        描述同一个路径下不同http操作对应的描述
    消息体内容/操作对象:
        是request和response对象都可以用上的对象
    操作对象中的参数和载荷:
        openapi提供两种机制来指定输入数据，parameters或者request body(消息载荷),参数
        parameters字段的值是一个数组，其中每个数组元素又是一个parameter object（单个的parameter对象),可以定义在path item对象那一级上也可以定义在定义在具体的操作对象上(operation object)
        介绍几个parameter中的基本字段:
            name: 参数名称 in: 参数所处的位置(path是路径的一部分,query是url中的查询参数,header: 在头信息中)，大多数情况下，字段的类型都通过在schema对象中指定,也就是增加一个schema字段，该schema字段的值就是一个schema对象用于指定该字段的信息,如果想要更高级的描述的话，那么可以通过content字段来指定,注意: schema和content字段二者只能存在其一
    复用描述对象:
        如果有多个相同的对象希望复用，可以使用根级结构下(也就是和info paths为同一级结构的)components来记录需要复用的部分
    api server(api服务器):
        在根级节点上通过servers来指定服务器的信息，也可以在path item,operation对象中指定，当多级节点下都指定了servers的时候，只有最低一级的生效
    
{
    paths: {
        "/users": {
            "get": {}
            servers: {
                url: ""
            }
        }
    }
    servers: {
        url: ""
    }
}
    服务器对象(server object)中还可以带变量，通过variables指定，为什么呢，因为服务器url中的有些字段的情况可能不止一种

这是OAS的官方网站: https://oai.github.io/Documentation/specification-servers.html

Swagger:
OAS(OpenApi Specification)和Swagger的关系:
    Swagger是提供调试、生成一系列符合OAS的api的工具，在谈论oas的时候实际上完全可以脱离swagger来谈论，因为swagger只是工具而已，而且Swagger（来自SmartBear）这个项目原本就是OAS的维护者，之前是由Swagger工具和Swagger规范组成的。只是后来将其移交给linux开源基金会了

Swagger也提供了一个代码生成工具叫做swagger codegen，看起来和openapi-generator差不多，有何区别?
    官方也答复了:Swagger Codegen是由SmartBear公司驱动的，而Openapi Generator是由社区驱动的，性质不一样，有超过40个来自Swagger Codegen团队的顶尖贡献者何模板创建者都加入到了OpenApi Generator团队中来，等于说Swagger是有商业性质的，而OpenApi Generator是完全开源民间性质的，对于开源开发者来说，使用OpenApi Generator就好了
