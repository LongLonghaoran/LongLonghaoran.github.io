Deployment:
kind为deployment就是deployment类型
主要有3个字段:
kind, metadata,spec
kind: deployment
metadata: 该deployment的元信息
spec.replicas: pod的副本数量
template: pod的spec内容,跟pod的差不多

kubectl edit deployment/pod可以通过编辑器直接修改资源的元数据，比如pod的副本数、镜像等
有一些快捷命令行操作，比如
kubectl set image deployments/deploymentName containerName=镜像名,containerName=镜像名, ...可以设置多个容器的镜像

Pod实际是由ReplicaSet管理的，replicaset中存储了和deployment类似的信息

而replicaSet的owner才是deployment,也就是说deployment是管理的replicaset,把在deployment.spec声明的关于pod的信息再给到replicaset来让其生成pod
deployment每次变更pod(比如变更某个容器的镜像)的时候，其实就是会生成新的replicaset,新的replicaset中存储就是变更后的deloyment中的内容,并且会保留之前的replicaset,
这样deployment在回滚的时候其实就是恢复上一个replicaset而已, 保留的ReplicaSet的数量限制也可以在deloyment中配置

每个ReplicaSet也就是对应了Deployment的一个版本，这点其实从ReplicaSet和Pod的名字也可以体现出来
ReplicaSet的名字是deploymentName-template-hash而pod的名字是deploymentName-template-hash-randomsuffix
其中的template-hash就是deployment中的spec.template所计算得出hash值，而一般对deployment进行版本升级也就是更改这里（更改镜像的版本）

Deployement控制器 & ReplicaSet 控制器原理
首先，我们所有的控制器都是通过 Informer 中的 Event 做一些 Handler 和 Watch。这个地方 Deployment 控制器，其实是关注 Deployment 和 ReplicaSet 中的 event，收到事件后会加入到队列中。而 Deployment controller 从队列中取出来之后，它的逻辑会判断 Check Paused，这个 Paused 其实是 Deployment 是否需要新的发布，如果 Paused 设置为 true 的话，就表示这个 Deployment 只会做一个数量上的维持，不会做新的发布。

如上图，可以看到如果 Check paused 为 Yes 也就是 true 的话，那么只会做 Sync replicas。也就是说把 replicas sync 同步到对应的 ReplicaSet 中，最后再 Update Deployment status，那么 controller 这一次的 ReplicaSet 就结束了。
那么如果 paused 为 false 的话，它就会做 Rollout，也就是通过 Create 或者是 Rolling 的方式来做更新，更新的方式其实也是通过 Create/Update/Delete 这种 ReplicaSet 来做实现的。
