概述

kylinfs csi driver 基于 juicefs csi driver v0.23.0构建，使用MountPod的挂载方式，方便csi controller 和csi node 配置更新和升级。

当前kylinfs 不支持目录quota 功能，所以 kylinfs csi driver  分配出来的存储卷实际上不具备容量限制功能。

kylinfs CSI driver 架构


juicefs提供了  Mount Pod、SideCar、进程挂载的三种架构，推荐使用Mount Pod的架构
[图片/附件]




说明：
  • 主要有CSI controller (statfulset)、CSI NodeService(daemonset)两个组件
  • 使用流程：
    • 用户创建 PVC，指定已经创建好的 StorageClass；
    • CSI Controller 负责在 JuiceFS 文件系统中做初始化，默认以 PV ID 为名字创建子目录，同时创建对应的 PV。该过程所需的配置，都在 StorageClass 中指定或引用；
    • Kubernetes (PV Controller 组件) 将上述用户创建的 PVC 与 CSI Controller 创建的 PV 进行绑定，此时 PVC 与 PV 的状态变为「Bound」；
    • 用户创建应用 Pod，声明使用先前创建的 PVC；
      • CSI Node Service 负责在应用 Pod 所在节点创建 Mount Pod；
    • Mount Pod 启动，执行 JuiceFS 客户端挂载，将挂载点暴露给宿主机，路径为 /var/lib/juicefs/volume/[pv-name]；
    • CSI Node Service 等待 Mount Pod 启动成功后，将 PV 对应的 JuiceFS 子目录 bind 到容器内，路径为其声明的 VolumeMount 路径；
    • Kubelet 启动应用 Pod。
优势：
  • 多个 Pod 共用 PV 时，不会新建 Mount Pod，而是对已有的 Mount Pod 做引用计数，计数归零时删除 Mount Pod。
  • CSI 驱动组件与客户端解耦，方便 CSI 驱动自身的升级

kylinfs 存储使用方法

kylinfs csi driver有静态pv和动态分配两种使用方法


静态存储 

  1. 需要手动创建PV
  2. pv创建完后，通过kylinfs-csi-driver挂载


[root@kfs1 static]# cat pv.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: kylinfs-pv
  labels:
    juicefs-name: ten-pb-fs
spec:
  # 目前 JuiceFS CSI 驱动不支持给静态 PV 设置存储容量，填写任意有效值即可
  capacity:
    storage: 1Gi
  volumeMode: Filesystem
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  mountOptions:
    - subdir=/app1
  csi:
    # 在先前的安装步骤中，已经创建了名为 csi.juicefs.com 的 CSIDriver
    driver: csi.kylinfs.com
    # volumeHandle 需要保证集群内唯一，因此一般直接用 PV 名即可
    volumeHandle: kylinfs-pv
    fsType: kylinfs
    # 在先前的步骤中已经创建好文件系统认证信息（Secret），在这里引用
    # 如果要在静态配置下使用不同的认证信息，甚至使用不同的 JuiceFS 文件系统，则需要创建不同的 Secret
    nodePublishSecretRef:
      name: kylinfs-secret
      namespace: default
 
 
[root@kfs1 static]# cat pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: kylinfs-pvc
  namespace: default
spec:
  accessModes:
    - ReadWriteMany
  volumeMode: Filesystem
  # 静态配置下必须指定 storageClassName 为空字符串
  # 代表该 PV 不采用任何 StorageClass，而是直接使用 selector 所指定的 PV
  storageClassName: ""
  # 由于目前 JuiceFS CSI 驱动不支持给静态 PV 设置存储容量，此处 requests.storage 填写任意小于 PV capacity 的有效值即可
  resources:
    requests:
      storage: 1Gi
  selector:
    matchLabels:
      juicefs-name: ten-pb-fs
 
 
 
[root@kfs1 static]# cat pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: kylinfs-app
  namespace: default
spec:
  containers:
  - args:
    - -c
    - while true; do echo $(date -u) >> /data/out.txt; sleep 5; done
    command:
    - /bin/sh
    image: arm64v8/debian:buster-slim
    imagePullPolicy: IfNotPresent
    name: app
    volumeMounts:
    - mountPath: /data
      name: data
    resources:
      requests:
        cpu: 10m
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: kylinfs-pvc

动态存储：StorageClass + PVC

可以实现动态供给，只需声明pvc，在pod调度完成后，容器创建时挂载kylinfs，无需管理员手动创建PV


[root@kfs1 dynamic]# cat pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: kylinfs-pvc-dynamic
  namespace: default
spec:
  accessModes:
  - ReadWriteMany
  resources:
    requests:
      # 从 StorageClass 中申请 1GiB 存储容量
      storage: 1Gi
  storageClassName: kylinfs-sc
    
 
[root@kfs1 dynamic]# cat storageclass.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: kylinfs-sc
provisioner: csi.kylinfs.com
parameters:
  csi.storage.k8s.io/provisioner-secret-name: kylinfs-secret
  csi.storage.k8s.io/provisioner-secret-namespace: default
  csi.storage.k8s.io/node-publish-secret-name: kylinfs-secret
  csi.storage.k8s.io/node-publish-secret-namespace: default
  kylinfs/mount-image: harbor.kylincloudnative.com/kylinfs/kylindata/mount-aarch64:latest
mountOptions:
  - no-syslog
  - no-usage-report
  - backup-meta=0
  - cache-size=10240
  - debug  
 
 
[root@kfs1 dynamic]# cat pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: kylinfs-app-dynamic
  namespace: default
spec:
  containers:
  - args:
    - -c
    - while true; do echo $(date -u) >> /data/out.txt; sleep 5; done
    command:
    - /bin/sh
    image: arm64v8/debian:buster-slim
    imagePullPolicy: IfNotPresent
    name: app
    volumeMounts:
    - mountPath: /data
      name: kylinfs-pv
  volumes:
  - name: kylinfs-pv
    persistentVolumeClaim:
      claimName: kylinfs-pvc-dynamic

kylinfs CSI driver 挂载传播分析


kylinfs csi driver 在pod对应节点 创建MountPod，并在MountPod 内部挂载kylinfs， 通过mount PROPAGATION 将挂载内容传递到宿主机，再通过bind 挂载将宿主机对应的目录绑定到pod的卷目录，使得pod可以访问MountPod挂载的目录。



MountPod 将挂载点暴露给宿主机

mount pod首先挂载了一个宿主机目录到容器内/var/lib/kylinfs/volume/， 并将mountPropagation设置为Bidirectional； Bidirectional相当于linux mount模式中的shard，意思是针对这个挂载点子目录如果有新的挂载，会被传递到另一边； 比如容器内在该目录下创建了一个子目录并挂载，挂载信息会被传递到宿主机；宿主机上同理

    securityContext:
      privileged: true   #使用Bidirectional必须设置为特权模式
      runAsUser: 0
...
 
    volumeMounts:
    - mountPath: /jfs
      mountPropagation: Bidirectional
      name: jfs-dir

宿主机将挂载点传递到应用pod

宿主机上执行bind挂载可以将一个目录的内容覆盖另外一个目录

spec:
  containers:
  - args:
    - -c
    - while true; do echo $(date -u) >> /data/out.txt; sleep 5; done
    command:
    - /bin/sh
    image: harbor.kylincloudnative.com/kylinfs/kylindata/kylinfs-base-aarch64:latest
    imagePullPolicy: IfNotPresent
    name: app
    resources: {}
    terminationMessagePath: /dev/termination-log
    terminationMessagePolicy: File
    volumeMounts:
    - mountPath: /data
      name: kylinfs-pv
      subPath: app2
    - mountPath: /var/run/secrets/kubernetes.io/serviceaccount
      name: kube-api-access-dzn9z
      readOnly: true  
  volumes:
  - name: kylinfs-pv
    persistentVolumeClaim:
      claimName: kylinfs-pvc-dynamic
 
 
 
pod被调度到节点上时，节点上的 kubelet 组件就会为这个 Pod 创建它的 Volume 目录，默认情况下 kubelet 为 Volume 创建的目录在 kubelet 工作目录下面：
 
/var/lib/kubelet/pods/<Pod的ID>/volumes/kubernetes.io~<Volume类型>/<Volume名字>
 
/var/lib/kubelet/pods/20056508-cda5-4f1b-b8c5-0c519745b761/volumes/kubernetes.io~csi/pvc-7ae0864f-b029-4559-a0b8-53acc8353209/mount
 
 
在宿主机上 在pod启动之前 使用mount --bind  /var/lib/kylinfs/volume/pvc-7ae0864f-b029-4559-a0b8-53acc8353209-zewfau /var/lib/kubelet/pods/20056508-cda5-4f1b-b8c5-0c519745b761/volumes/kubernetes.io~csi/pvc-7ae0864f-b029-4559-a0b8-53acc8353209/mount
 
 
kubelet在启动容器时会执行类似的命令
docker run -v /var/lib/kubelet/pods/<Pod的ID>/volumes/kubernetes.io~<Volume类型>/<Volume名字>:/<容器内的目标目录> IMAGE ...
 
将暴露到宿主机上的kylinfs 挂载点绑定到pod的volume内，pod启动完成后pod内就能在看到kyinfs挂载点了
