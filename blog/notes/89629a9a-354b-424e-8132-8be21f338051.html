1. Alluxio介绍

Alluxio是一种分布式缓存系统，属于数据编排的范畴，主要云端环境下的数据分析和AI场景。Alluxio是为了数据从底层存储更快的到的应用程序。Alluxio 统一了存储在不同存储系统中的数据，向其上层的数据驱动应用程序提供统一的客户端 API 和全局命名空间。

[图片/附件]

优点：

  • 内存速度的 I/O：Alluxio 可以用作分布式共享缓存服务，使得与 Alluxio 通信的计算应用程序可以透明地缓存频繁访问的数据，特别是来自远程位置的数据，从而提供内存级的 I/O 吞吐量。此外，Alluxio 的分层存储能够同时利用内存和磁盘（SSD/HDD），使得数据驱动的应用程序在弹性扩展时具有成本效益。
  • 简化云和对象存储的使用：云和对象存储系统使用不同的语义，这些语义相比传统文件系统会对性能产生影响。常见的文件系统操作，如目录列表和重命名，通常会带来明显的性能开销。在访问云存储中的数据时，应用程序没有节点级的本地性或跨应用程序缓存。Alluxio 与云或对象存储联合使用，通过从 Alluxio 而非底层云或对象存储中提供数据，可以带来性能提升。
  • 简化数据管理：Alluxio 提供了一个访问多个数据源的单一入口。除了连接不同类型的数据源外，Alluxio 还允许用户同时连接同一存储系统的不同版本，如多个版本的 HDFS，而无需复杂的系统配置和管理。
  • 简化应用程序部署：Alluxio 管理应用程序与文件或对象存储之间的通信，将应用程序的数据访问请求转换为访问底层存储接口。Alluxio 与 Hadoop 兼容，现有的数据分析应用程序，如 Spark 和 MapReduce 程序，可以在 Alluxio 之上运行而无需任何代码修改。

创新点：

  • 全局命名空间：Alluxio 作为多个独立存储系统的单一访问点，不论其物理位置如何。这为所有数据源提供了统一视图和标准接口。
  • 智能多层缓存：Alluxio 集群作为连接存储系统中数据的读写缓存。可配置的策略会自动优化数据在内存和磁盘（SSD/HDD）上的放置，以提高性能和可靠性。缓存对用户是透明的，使用缓冲区来保持与持久存储的一致性。
  • 服务器端 API 转换：Alluxio 支持行业通用的 API，如 HDFS API、S3 API、FUSE API、REST API。它透明地将标准客户端接口转换为任何存储接口。Alluxio 管理应用程序与文件或对象存储之间的通信，消除了复杂系统配置和管理的需求。文件数据可以看起来像对象数据，反之亦然。


2. Alluxio架构

Alluxio 在大数据和机器学习生态系统中作为一个新的数据访问层，位于持久存储系统（如 Amazon S3、Microsoft Azure Object Store、Apache HDFS 或 OpenStack Swift）与计算框架（如 Apache Spark、Presto 或 Hadoop MapReduce）之间。Alluxio 本身并不是一个持久存储系统。将 Alluxio 用作数据访问层可以提供多种好处：

  • 对于用户应用程序和计算框架，Alluxio 提供快速存储，并在不同应用程序之间实现数据共享和本地化。因此，当数据存在 Alluxio 中时，可以以内存速度提供数据；当数据不在 Alluxio 中时，可以以计算集群网络的速度提供数据。首次访问时，数据仅从底层存储系统读取一次。当访问底层存储的速度较慢时，数据访问速度显著加快。为了实现最佳性能，建议将 Alluxio 与集群的计算框架一起部署。
  • 对于底层存储系统，Alluxio 实现了应用程序和传统存储系统之间的桥梁，扩展了使用数据的工作负载范围。Alluxio之上的应用程序可以使用任何Alluxio之下的底层存储。同时挂载多个底层存储系统时，Alluxio 作为统一层，可以整合不同的数据源。

[图片/附件]

Alluxio 分为三个组件：masters、workers、clients。一个典型的集群包括leading master、standby masters、job master、standby job masters、workers和job workers。master和worker进程构成了 Alluxio 服务器，这是系统管理员需要维护的组件。客户端用于应用程序（如 Spark 或 MapReduce 作业）、Alluxio 命令行界面（CLI）或 Alluxio FUSE 层与 Alluxio 服务器进行通信。

Alluxio job masters和job workers可以分离为一个独立的功能，称为作业服务（Job Service）。Alluxio  job service是一个轻量级的任务调度框架，负责向job workers分配不同类型的操作。这些操作包括：

  • 从 UFS（底层文件系统）加载数据到 Alluxio
  • 将数据持久化到 UFS
  • 在 Alluxio 内部复制文件
  • 在 UFS 或 Alluxio 位置之间移动或复制数据

job service的设计使得所有与作业相关的进程不一定需要与 Alluxio 集群的其他部分位于同一节点。但是，建议将job workers与对应的 Alluxio worker放在同一节点，以降低 RPC 和数据传输的延迟。



3. Masters

[图片/附件]

Alluxio 包含两种master进程。一个是 Alluxio master。Alluxio master负责处理所有用户请求，并记录文件系统元数据的变更。另一个是 Alluxio  job master，这个进程作为一个轻量级调度器，调度文件系统操作，然后这些操作由 Alluxio job worker执行。

Alluxio master可以部署为一个leading master和多个sandby master以实现故障容错。当leading master出现故障时，会选举一个备用主节点成为新的leading master。



3.1. leading masters

在 Alluxio 集群中，只有一个master进程可以作为leading master。leading master负责管理集群的全局元数据，包括文件系统元数据（如文件系统的 inode 树）、块元数据（如块的位置）以及工作节点的容量元数据（空闲和已用空间）。leading master只会查询底层存储的元数据，应用数据不会通过master路由。Alluxio 客户端通过leading master来读取或修改元数据。所有worker定期向leading master发送心跳信息。leading master不会主动与其他组件通信；它只通过 RPC 服务响应请求。leading master将所有文件系统事务记录到一个分布式持久存储位置，以便在需要时恢复master的状态信息；这些记录的集合称为日志（journal）。



3.2. standby masters

standby master部署在不同的服务器上，以在 Alluxio 运行在高可用（HA）模式下提供故障容错功能。standby master通过读取leading master写入的日志（journal），保持master副本的更新。standby master还会写入日志检查点，以便更快恢复。但是，standby master不会处理来自其它 Alluxio 组件的任何请求。在leading master发生故障切换后，备用主节点重新选举新的leading master。



3.3. secondary masters（用于UFS日志）

当在没有高可用（HA）模式的单一 Alluxio  master下运行，并使用 UFS 日志时，可以在与leading master相同的服务器上启动一个secondary master，用于写入日志检查点。secondary master并不是为了提供高可用性而设计的，而是为了减轻leading master的工作负担，以实现更快的恢复。与standby master不同，secondary master不会升级为leading master。



3.4. job masters

Alluxio job master是一个独立的进程，负责在 Alluxio 内异步调度一系列较重的文件系统操作。通过将这些操作与leading master的执行分离，Alluxio maser可以减少资源消耗，从而在更短的时间内服务更多的客户端。此外，它还提供了一个可扩展的框架，以便将来可以添加更复杂的操作。

Alluxio job master接收执行上述操作的请求，并将这些操作调度到 Alluxio  job worker上执行（job worker作为 Alluxio 文件系统的客户端）。



4. Workers

[图片/附件]



4.1. Alluxio workers

Alluxio workers负责管理分配给 Alluxio 的本地资源（例如内存、SSD、HDD）。Alluxio worker将数据存储为块，并通过读取或在本地资源中创建新块来服务客户端的读写请求。worker仅负责管理块；文件到块的实际映射仅由master存储。

worker在底层存储系统上执行数据操作。有两个重要的好处：

  1. 从底层存储系统读取的数据可以存储在工作节点中，并立即提供给其它客户端。
  2. 客户端可以是轻量级的，不依赖于底层存储连接。

由于 RAM 通常提供的容量有限，当空间满时，worker中的块可能会被驱逐。worker采用驱逐策略来决定在 Alluxio 空间中保留哪些数据。



4.2. Alluxio job workers

Alluxio job worker是 Alluxio 文件系统的客户端。它们负责执行 Alluxio job master分配的任务。job worker接收指令，在指定的文件系统位置执行加载、持久化、复制、移动或复制操作。

Alluxio job worker不一定需要与worker共同放置，但建议将两者部署在同一物理节点上。



5. Client

Alluxio 客户端提供了与 Alluxio 服务器交互的网关。它会与leading master进行通信以执行元数据操作，并与worker进行通信以读取和写入存储在 Alluxio 中的数据。Alluxio 支持 Java 的本地文件系统 API，以及 REST、Go 和 Python 等。Alluxio 还支持与 HDFS API 和 Amazon S3 API 兼容的 API。

Alluxio 客户端不会直接访问底层存储系统。数据通过 worker进行传输。



6. 几种读数据流场景



6.1. 本地缓存命中

发生本地缓存命中时，请求的数据驻留在本地的 Alluxio  worker上。例如，如果一个应用程序通过 Alluxio 客户端请求数据访问，客户端会向 Alluxio master询问数据所在的worker位置。如果数据是本地可用的，Alluxio 客户端会使用“短路”读取直接通过本地文件系统读取文件，绕过 Alluxio worker。短路读取避免了通过 TCP 套接字进行数据传输，并直接提供数据访问。这是从 Alluxio 中读取数据的最快方式。

默认情况下，短路读取使用需要宽松权限的本地文件系统操作。当worker和client容器化时，有时是无法实现的，因为资源计量不正确。在默认的短路读取不可行的情况下，Alluxio 提供基于域套接字的短路读取，其中worker通过预定的域套接字路径将数据传输到客户端。

另外需要注意的是，Alluxio 除了内存之外，还可以管理其他存储介质（如 SSD、HDD），因此本地数据访问速度可能因本地存储介质的不同而有所不同。



6.2. 远程缓存命中

当请求的数据存储在 Alluxio 中，但不在客户端的本地worker节点上时，客户端会从持有该数据的远程worker节点执行读取操作。在客户端读取完数据后，它会指示本地worker节点（如果存在）在本地创建一份副本，以便将来对相同数据的读取可以在本地完成。远程缓存命中提供网络速度的数据读取。Alluxio 优先从远程worker节点读取，而不是从底层存储读取，因为 Alluxio worker节点之间的网络速度通常比 Alluxio worker节点与底层存储之间的速度更快。



6.3. cache丢失

如果数据在 Alluxio 中不可用，就会发生缓存未命中，应用程序需要从底层存储读取数据。Alluxio 客户端将从 UFS（底层文件系统）读取的任务委派给worker节点，优先选择本地worker节点。这个worker节点从底层存储读取数据并缓存。缓存未命中通常会导致最大的延迟，因为数据必须从底层存储获取。第一次读取数据时通常会发生缓存未命中。

当客户端仅读取一个块的一部分或非顺序地读取该块时，客户端会指示worker节点异步缓存整个块，这被称为异步缓存。异步缓存不会阻塞客户端，但如果 Alluxio 和底层存储系统之间的网络带宽是瓶颈，可能会影响性能。



6.4. cache跳过

可以通过在客户端中将属性 alluxio.user.file.readtype.default 设置为 NO_CACHE 来关闭 Alluxio 中的缓存功能。



7. 几种数据流写方式



7.1. 只写到Alluxio（MUST_CACHE）

写入类型为 MUST_CACHE 时，Alluxio 客户端仅将数据写入本地 Alluxio 工作节点，而不会将数据写入底层存储。在写入过程中，如果短路写入可用，Alluxio 客户端会直接将数据写入本地 RAM 磁盘中的文件，绕过 Alluxio 工作节点以避免网络传输。由于数据不会持久化到底层存储，如果机器崩溃或需要释放空间以进行新的写入，数据可能会丢失。MUST_CACHE 设置适用于在可以容忍数据丢失的情况下写入临时数据。



7.2. 同步写入UFS（CACHE_THROUGH）

当写入类型为 CACHE_THROUGH 时，数据会同步写入 Alluxio 工作节点和底层存储系统。Alluxio 客户端将写入任务委托给本地工作节点，该工作节点同时将数据写入本地内存和底层存储。由于底层存储通常比本地存储写入速度慢，客户端的写入速度将匹配底层存储的写入速度。CACHE_THROUGH 写入类型在需要数据持久化时推荐使用。数据还会在本地存储一份副本，以便将来对该数据的读取可以直接从本地内存中提供。



7.3. 异步写入UFS（ASYNC_THROUGH）

Alluxio 提供了一种 ASYNC_THROUGH 的写入类型。使用 ASYNC_THROUGH 时，数据首先同步写入到 Alluxio 工作节点，然后在后台持久化到底层存储。ASYNC_THROUGH 可以提供接近 MUST_CACHE 的数据写入速度，同时仍然能够持久化数据。自 Alluxio 2.0 起，ASYNC_THROUGH 是默认的写入类型。

为了提供容错能力，与 ASYNC_THROUGH 一起使用的一个重要属性是 alluxio.user.file.replication.durable。该属性设置了在写入完成后但在数据持久化到底层存储之前，Alluxio 中新数据的目标副本级别，默认值为 1。在后台持久化过程完成之前，Alluxio 将维持文件的目标副本级别，然后回收 Alluxio 中的空间，因此数据只会写入到底层存储一次。

如果在使用 ASYNC_THROUGH 写入副本时，所有包含副本的工作节点在数据持久化之前崩溃，那么数据将会丢失。



7.4. 只写到UFS（THROUGH）

使用 THROUGH 写入类型时，数据会写入到底层存储，而不会缓存到 Alluxio 工作节点中。此写入类型确保数据在写入完成之前被持久化，但速度受限于底层存储的吞吐量。



7.5. 数据一致性

无论写入类型如何，Alluxio 空间中的文件/目录始终是强一致的，因为所有这些写操作都会先通过 Alluxio master ，修改 Alluxio 文件系统之后，才返回成功状态给客户端/应用程序。因此，只要相应的写操作成功完成，不同的 Alluxio 客户端总是会看到最新的更新。

然而，对于将 UFS 中的状态考虑在内的用户或应用程序，不同写入类型下可能会有不同的表现：

  • MUST_CACHE 不会将数据写入 UFS，因此 Alluxio 与 UFS 不会一致。
  • CACHE_THROUGH 会在返回成功给应用程序之前，同步将数据写入 Alluxio 和 UFS。
    • 如果写入 UFS 也是强一致的（例如 HDFS），只要 UFS 中没有其他外部更新，Alluxio 将与 UFS 一致；
    • 如果写入 UFS 是最终一致的（例如 S3），可能会出现文件成功写入 Alluxio 但稍后才出现在 UFS 中的情况。在这种情况下，Alluxio 客户端仍将看到一致的文件系统状态，因为它们总是咨询强一致的 Alluxio master；因此，尽管不同的 Alluxio 客户端仍然在 Alluxio 空间中看到一致的状态，但在数据最终传播到 UFS 之前可能会有一个不一致的窗口。
  • ASYNC_THROUGH 将数据写入 Alluxio 后返回给应用程序，Alluxio 异步地将数据传播到 UFS。从用户的角度来看，文件可以成功写入 Alluxio，但稍后会在 UFS 中持久化。
  • THROUGH 直接将数据写入 UFS 而不在 Alluxio 中缓存数据，然而，Alluxio 知道文件及其状态。因此，元数据仍然是一致的。


8. Alluxio分层存储

Alluxio支持分层存储，以便管理内存之外的其它存储类型。目前Alluxio支持如下存储类型(存储层)：

  • MEM (内存)
  • SSD (固态硬盘)
  • HDD (硬盘驱动器)

使用分层存储使得Alluxio可以在系统中存储更多数据，因为内存容量在一些部署中非常有限。有了分层存储，Alluxio自动管理存放于不同存储层中的数据块，因此用户和管理员无需手动管理数据存放的位置。用户也可以通过实现分配策略和回收策略指定自己的数据管理策略。



8.1. 使用分层存储

引入分层存储后，Alluxio管理的数据块不只在内存中，可存放于任何可用的存储层。Alluxio使用分配策略和回收策略管理块的存放和移动。Alluxio根据I/O性能的高低从上到下配置存储层。因此，这种配置策略决定了最顶层存储是MEM，然后是SSD，最后是HDD。



8.1.1. 存储目录

一个存储层至少有一个存储目录。目录是Alluxio数据块存放的文件路径。Alluxio支持单个存储层包含多个目录的配置，允许一个存储层有多个挂载点或存储设备。举例而言，如果Alluxio worker上有5个SSD设备，可以配置Alluxio在SSD层同时使用这5个SSD设备。分配策略决定数据块文件存放的目录。



8.1.2. 写数据

用户写入新数据块时默认写在顶层存储(不想使用默认配置可以使用定制的分配策略)。如果顶层没有足够的空间存放数据块，回收策略会被触发并释放空间给新数据块。



8.1.3. 读数据

读取分层存储的数据块和标准Alluxio类似。Alluxio从存储位置读取数据块。如果Alluxio配置了多层存储，数据块不一定是从顶层读取，因为可能被透明地移到下层存储中。

读取策略为AlluxioStorageType.PROMOTE时，Alluxio会确保数据在读取前先被移动到顶层存储中。通过显式的将热数据移到最高层,该策略也可以用于数据块的管理。



8.1.4. 固定文件

另一控制文件存放和移动的方法是固定(pin)和取消固定(unpin)文件。文件被固定时，数据块不会从Alluxio的存储空间中移出。同时用户可以将固定文件的数据块移到顶层存储。

由于已固定文件的数据块不会被移出，client要确保在合适的时候取消固定文件。



8.1.5. 分配策略

Alluxio使用分配策略选择新数据块的写入位置。Alluxio定义了分配策略的框架，也内置了几种分配策略。以下是Alluxio已实现的分配策略：

  • 贪心分配策略

    分配新数据块到首个有足够空间的存储目录。

  • 最大剩余空间分配策略

    分配数据块到有最大剩余空间的存储目录。

  • 轮询调度分配策略

    分配数据块到有空间的最高存储层，存储目录通过轮询调度选出。



8.1.6. 回收策略

Alluxio使用回收策略决定当空间需要释放时，哪些数据块被移到低存储层。Alluxio支持自定义回收策略，已有的实现包括：

  • 贪心回收策略

    移出任意的块直到释放出所需大小的空间。

  • LRU回收策略

    移出最近最少使用的数据块直到释放出所需大小的空间。

  • LRFU回收策略

    基于权重分配的最近最少使用和最不经常使用策略移出数据块。如果权重完全偏向最近最少使用,LRFU回收策略退化为LRU回收策略。

  • 部分LRU回收策略

    基于最近最少使用移出，但是选择有最大剩余空间的存储目录(StorageDir)，只从该目录移出数据块。

使用同步移出时，推荐使用较小的块大小配置（64MB左右），以降低块移出的延迟。使用空间预留器时，块大小不会影响移出延迟。



8.1.7. 空间预留器

空间预留器可以在存储空间被完全耗尽前试着在每一层预留一定比例的空间。用于改善突发写入的性能，也可以在回收策略连续运行时提高持续写入的边际性能增益。



8.2. 开启和配置空间预留

在Alluxio中，使用配置参数开启分层存储。默认情况下，Alluxio使用单层内存存储。使用如下配置参数可以指定Alluxio的额外存储层：

相关配置示例：

  • alluxio.worker.tieredstore.levels=2 在Alluxio中配置了两级存储
  • alluxio.worker.tieredstore.level0.alias=MEM配置了首层(顶层)是内存存储层
  • alluxio.worker.tieredstore.level0.dirs.path=/mnt/ramdisk 定义了/mnt/ramdisk是首层的文件路径
  • alluxio.worker.tieredstore.level0.dirs.quota=100GB设置了ramdisk的配额是100GB
  • alluxio.worker.tieredstore.level0.reserved.ratio=0.2设置了顶层的预留空间比例是0.2
  • alluxio.worker.tieredstore.level1.alias=HDD配置了第二层是硬盘驱动器层
  • alluxio.worker.tieredstore.level1.dirs.path=/mnt/hdd1,/mnt/hdd2,/mnt/hdd3配置了第二层3个独立的文件路径
  • alluxio.worker.tieredstore.level1.dirs.quota=2TB,5TB,500GB定义了第二层3个文件路径各自的配额
  • alluxio.worker.tieredstore.level1.reserved.ratio=0.1设置了第二层的预留空间比例是0.1

定义存储层时有一些限制。首先，最多只有3个存储层。而且一层最多指定一个别名。举例而言，如果想在HDD层使用多个硬盘驱动器，可以配置alluxio.worker.tieredstore.level{x}.dirs.path为多个存储路径。

另外，可以配置特定的回收策略和分配策略。

空间预留可以配置为开启或关闭：

alluxio.worker.tieredstore.reserver.enabled=false
