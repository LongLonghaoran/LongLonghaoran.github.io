写流程

  1. 用户通过posix 接口发起写请求
  2. 客户端查询元数据信息获取写对象属性信息，文件是否存在，文件大小等信息
  3. 客户端实时获取cache server 存活节点情况，根据cache server数量和block key，通过一致性hash计算block所在cache server节点
    1. 同步执行数据缓存
    2. 异步向后端PUT数据
  4. cache sever 数据缓存完成则返回客户端
    1. cache写完就返回会破坏元数据中记录信息的一致性（元数据记录了block后端实际不存在）
    2. 可以根据用户一致性配置 决定时写完后端再返回还是写完cache就返回
  5. client 收到后段put成功后，更新元数据信息
  6. 如果client 重试写cache多次仍失败，则直接写后端。
  7. 多个客户端并发写同一文件时不保证一致性；

cache server扩缩容

cache server 根据一致性hash扩容

扩容流程：

  1. 管理员后台直接增加cache server 节点
  2. 使用warmup命令重新缓存最近读取的数据
  3. 用户发起新的读写请求
  4. client 通过一致性hash重新计算block所属cache server，如果cache server上没有缓存block数据则从后台读取并重新缓存； 
  5. 根据一致性hash计算规则，cache server节点上不属于本节点的缓存数据，可通过以下方式处理
    1. 缓存过期策略淘汰
    2. 定期扫描cache server上不属于本节点的block缓存数据并清理


缩容流程：

  1. 管理员后台禁用cache server 节点
  2. 使用warmup命令重新缓存最近读取的数据
  3. 用户发起新的读写请求
  4. client 通过一致性hash重新计算block所属cache server，如果cache server上没有缓存block数据则从后台读取并重新缓存； 
  5. 根据一致性hash计算规则，cache server节点上不属于本节点的缓存数据，可通过以下方式处理
    1. 缓存过期策略淘汰
    2. 定期扫描cache server上不属于本节点的block缓存数据并清理
  6. 管理员后台删除cache server节点



实现细节：

  1. cache server启动时自动注册到元数据服务中
  2. cache server被删除或者长期连不上，清理？
    1. 需要有后台进程处理探活逻辑，client数量多，不适合； cache server互相探活在网络分区时可能异常
    2. 通过命令执行cache server删除逻辑较为合适

alluxio的一致性

Alluxio 与底层存储系统之间的一致性保证：这部分的一致性较为复杂，取决于配置和底层存储系统的类型：

  • Must_Cache 模式：这种模式下，数据只会缓存在 Alluxio 中，不会写入到底层存储，所以 Alluxio 空间与底层存储是不一致的。这种模式适用于对数据一致性要求不高，更注重读取性能的场景。
  • Cache_Through 模式：在成功返回应用程序之前，会将数据同步写入 Alluxio 和底层存储中。如果底层存储是 HDFS，由于 HDFS 本身的写入也是强一致性的，所以当底层存储中没有其他绕过 Alluxio 的直接更新时，Alluxio 空间将始终与底层存储保持一致；如果底层存储是 S3 等对象存储，写入底层存储是最终一致性的，有可能文件已成功写入 Alluxio，但不会立即显示在底层存储中，会产生一个不一致的短期窗口1。
  • Async_Through 模式：数据先写入 Alluxio 并返回到应用程序，然后由 Alluxio 将数据异步传播到底层存储。从用户的角度来看，文件可以成功地写入 Alluxio，但是会滞后持久保存到底层存储1。
  • Through 模式：数据直接写入底层存储，而不在 Alluxio 中缓存数据，但是 Alluxio 知道文件及其状态，所以元数据仍然是一致的。这种模式适用于对底层存储的直接写入操作较多，而对 Alluxio 的缓存需求不高的场景。

v2版本设计

v1版本倾向于快速验证cache功能的可用性；cache server和kylinfs使用统一的元数据服务，client使用chunkstore本地磁盘缓存模块来存储缓存数据。


该版本存在以下问题：

  1. 元数据统一，无法使用多UFS
  2. 未考虑client本地磁盘缓存和cache 缓存一致性问题； 也是cache元数据和kylinfs元数据一致性问题。
  3. cache 淘汰算法使用默认的逻辑； 未针对AI场景优化。
  4. 无client co-desgin。



cache 元数据设计

  1. 基于内容hash，client直接拉取hash列表文件； hash列表文件全放缓存；
  2. client 获得hash列表文件后，找到对应的hash key，读cache，cache存在则直接返回，不存在则代理向UFS读，读完后返回client并save cache
  3. client cache处理多用户并发。相同cache则复用


是否要借助于现有的元数据读写逻辑？

  1. cache 存储粒度用文件 还是 block？还是 基于内容的hash
  2. 用文件粒度太粗，如果加载一个大的ImageNet，可能直接就存在其中一个cache server，cache server容量不够，这样没有意义
  3. 用block存储，需要维护block元数据，metadata for cache 存储与UFS相同的元数据；无法确保每个UFS元数据都相同，所以不能用chunk-block形式的元数据
  4. 只能用一种通用的元数据存储格式，适合所有UFS的
