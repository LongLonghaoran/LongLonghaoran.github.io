Kubevirt VirtualMachines
笔记本：Kubevirt
Virtual machine Instances:
概念上由2部分组成: 调度决定的信息;虚拟机api的信息\
每个vmi对象代表耽搁运行中的虚拟机实例,可以通过kubectl与其像操作k8s对象一样操作它
生命周期
可以通过vmi.yaml文件kubectl apply手动创建
通过vm以及virtctl工具启动/停止/暂停/恢复vmi
运行策略
与spec.running互斥，二选一
always: vmi总是会存在.如果vmi死机了,新的就会创建。与spec.running: true行为一致
RerunonFailure:vmi会重新创建，如果先前的实例在错误状态下失败的话。它如果客户机成功终止的话（比如在虚拟机内部关机）是不会重新创建的
Manual:不论是vmi存在还是不存在，都是由手动控制，kubevirt不会自动转换它的状态
Halted: 总是会删除vmi实例，与spec.running: false一致
实例类型和偏好设置
vm api包含了用于调整vm性能的高级选项，但这超过典型用户的认知。用户无法用广泛的术语来简单地定义存储、网路
instancetypes和preferences提供了定义一套资源、性能和其他运行时指标的方式，允许用户在多个vm中复用这些定义。可以定义cpu memory gpus hostdevices等，其中cpu和memory是必填的，在vm中不可覆盖
preference就是用于定义偏好值的，可以在vm中覆盖，当vm中不指定的时候就使用偏好值，可以定义diskBus等值
在vm中通过spec.instancetype和spec.preference字段指定
预设Preset(v0.57.0之后已废弃，由instancetype和preference替代)

接口和网络 Interfaces and Networks
将vm连接到网络分为2个部分:1.通过spec.networks指定网络，然后通过指定spec.domain.devices.interfaces将这些网络支持的接口配置好.
每个接口必须由对应的同名的网络.
interface定义了vm的虚拟网络接口（也称为前端）。一个network指定了interface的后端并声明了它连接到逻辑或者物理设备（也称为后端）。
由多种方式配置interface和network。
后端backend:
在spec.networks中指定.network必须使用唯一的名称。每个网络都要指定一个类型type,可以是Pod: 默认的网络,multu使用multus提供的第二网络,也可通过在配置network时指定default: true来让multus称为唯一一个网络
前端frontend:
也即在虚拟机中可见的部分,也就是网卡,通过 spec.domain.devices.interfaces配置 。相同的网络network后端可能以多种不同的方式连接到虚拟机，每种有他们自己的连接保证和指标,
限制有以下集中可用类型: bridge, slirp, sriov, masquerade
multus文档:https://github.com/k8snetworkplumbingwg/multus-cni

Istio service mesh服务网格
kubevirt允许作为istio的一部分运行vm
限制: 只支持masquerade方式(NAT)
istio使用特定端口，这些端口不能被vmi使用
istio只支持ipv4

网络策略Network Policy:
k8s自带的资源Network Policy资源对象,为了使用network policy对象，必须先使用一个支持network policy的network解决方案.network isolation是被network policy对象控制的。默认情况下，同一命名空间下所有vmi都可以互相访问也可以被其他网络节点网络。为了独立一个或更多vmi。你可以创建网络策略对象来指示允许的进入连接。
注意：在网络策略中pod和vmi是同等对待的，因为labels被传递到pod中，而pod中含有运行着的vmi。换句话说，vmis上的labels截图通过网络策略的podselector匹配到

Guest Agent information
ga是一个运行在虚拟机中的可选的组件。ga提供了大量的运行时信息关于正在运行的操作系统。
当ga连接上vmi的时候，vmi上会有一条状态信息,也就是在status字段下会有一个type为AgentConnected的状态存在告知
当ga连接上之后，额外的os信息就会显示在status处。这个信息由以下组成:
1.客户机信息 2.接口信息，展示的是qemu接口

访问虚拟机Accessing Virtual Machine
https://kubevirt.io/user-guide/virtual_machines/accessing_virtual_machines/
两种方式vnc,ssh
vnc:通过virtctl vnc vm-name访问或者virtctl vnc --proxy-only vm-name,仅开放vnc端口，供其它机器远程连接
ssh:通过public key注入到虚拟机后可以使用virtctl ssh免密登录到虚拟机中
有2种注入方式:静态和动态
静态:  1.创建 secret，内容是本地的id_rsa.pub文件,2.创建虚拟机时指定accessCredentials字段,利用该字段配置sshPublicKey
动态: 通过qemu-guest-agent进行,需要先确保  qemu-guest-agent 安装好了
