1.关于forEach 和 $.each
    forEach是es5新增的方法，它在ie8以及以下的浏览器不支持
    each是jquery提供的方法，一般来说它已经做好了兼容

2.不要总是将url路径误认为是文件路径
    场景1：比如通过服务器去伺服一个文件夹，那么访问该服务的主机名域名根url的时候，对应的是文件夹的根目录，我们可以在Url后面补充路径，然后对应去访问被伺服的服务器文件夹下开始的其他文件。但是通过昨天的例子我们也知道这个完全是由程序去控制的。
    场景2：.html文件中会引入各类静态资源文件，比如 js css img等,需要注意的是其中的 src href 属性中的路径
也是值得url路径，而并非该资源在服务器文件夹上相对于该.html文件的位置

3.状态码 301 302
    301永久重定向,浏览器将记住这个重定向，比如由 a.com 重定向到 b.com, 如果是永久重定向的话，那么今后
    浏览器只要碰到 a.com请求都会自动的重定向到 b.com去
    302临时重定向，浏览器不会记住，这个重定向仅仅作用于当前这次请求，也就是说下次访问a.com的时候，只要服务端不继续响应重定向的话，那么浏览器还是会正常的 访问 a.com,而不会再次地重定向了
    
4.Node中的模块系统
    使用Node编写应用程序就是在使用：ECMAScript 语言、核心模块、第三方模块、咱们自己写的模块
    4.1 CommonJs模块规范
        1.js本身是不支持模块化的
        2.什么是模块化：
            具有文件作用域
            通信规则（加载与导出）
        3.commonJs模块规范
            模块作用域
            使用require 方法来加载模块
            使用exports 来导出变量
    4.2.nodeJs中的模块规则:
        1. 被require的模块可以通过exports对象将需要共享的数据共享出去，主动require的模块，则可以通过
            被引入进来的模块来获取这些变量,具体做法是:
//a.js 
var b = require('./b');
b.foo
//b.js
exports.foo = '这是要共享的值'

        2.如果希望require一个模块之后，可以直接使用这个模块，作为方法或者作为值来使用(比如art-template, 引入之后，直接使用template作为方法调用)，那么应该使用 module.exports = 的形式直接赋值来导出
//b.js
module.exports = 'hello'
//a.js
var b = require('b')
console.log(b) // hello

        3.exports和module.exports的区别
            1.在Node中，可以认为每个模块内部中都有一个自己的module对象
            2.在module中有一个成员/属性，叫做 exports
            3.在代码的最后有一句 return module.exports
            4.谁来require这个模块，那么谁就可以通过require方法获得这个返回值 module.exports
            5.需要导出多个内部成员的话，应该是多次使用module.exports.变量名 = xxx 的方式来导出
                    于是，nodejs为了简化操作就在模块内部增加了一个变量 exports 它是一个指向
                    module.exports的引用,也就是说 exports === module.exports
            6.但是，当直接使用 exports 或者 module.exports赋值的时候，情况却不一样： 直接给exports赋
                值是不管用的,这是由于 exports 和 module.exports他们都只是个引用而已
exports 和 module.exports 是指向同一个对象的引用，但是这两个变量本身是不关联的，如果我们直接修改exports的指向的话，那么module.exports是不受影响的
            7.一定要记住，模块内部最后return 的是 module.exports，而不是 exports
                也就是说 如果我们在模块内部 执行 exports = {}，那么之后再通过exports.变量名 的方式来赋值，
                这样将起不到共享数据的作用
                同理，如果执行 module.exports = xxx 直接将exports的指向更改了，那么它之后的 exports也会                   失效

            8.网络上的人的经验也验证了这一点
//来自简书
exports只能通过“.”的方式暴露属性和方法
eg:   exports.name = '张三'


module.exports可以通过“.”的方式或 = { }方式
eg: module.exports.name = "张三"
或  module.exports= {name: "张三"}
可以注意到：人家的经验表明，不要直接 给 exports赋值，也不要直接给 module.exports赋值,这样也就避免了另一个变量赋值的失效，当然我们现在已经知道, module.exports可以直接赋值，前提是这个模块只需要返回一个 变量/方法
        9.最后： 实在记不住上面的理论，你可以无视exports，当它不存在，只使用module.exports即可....
