Sameer Ajmani
2014年3月13日

介绍
Go的并发原语使得构建能够高效利用I/O和多个CPU的流水线式数据处理变得轻松。本文介绍了这种流水线的示例，突出了操作失败时出现的微妙之处，并介绍了处理失败的技术。

什么是流水线？
在Go中，对于流水线并没有正式的定义；它只是众多种类的并发程序之一。非正式地说，流水线是一系列由通道连接的阶段，每个阶段都是一组运行相同函数的goroutine。在每个阶段中，这些goroutine会：

通过入站通道从上游接收值
对数据执行某些函数，通常会生成新的值
通过出站通道将值发送到下游
每个阶段都可以有任意数量的入站和出站通道，除了第一个和最后一个阶段，它们分别只有出站或入站通道。第一个阶段有时被称为源头或生产者；最后一个阶段则称为汇聚点或消费者。

我们将从一个简单的示例流水线开始，以解释这些思想和技术。随后，我们将呈现一个更实际的示例。

平方数
考虑一个包含三个阶段的流水线。

第一个阶段是gen函数，它将整数列表转换为一个通道，该通道会发射列表中的整数。gen函数启动一个goroutine，将整数发送到通道，并在所有值都发送完毕时关闭通道：
func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}
第二阶段是sq，它从通道接收整数，并返回一个通道，该通道会发射每个接收到的整数的平方。在入站通道关闭后，且该阶段已将所有值发送到下游之后，它会关闭出站通道：
func sq(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}
主函数设置了流水线并运行了最终的阶段：它从第二阶段接收值，并打印每个值，直到通道关闭为止：
func main() {
    // Set up the pipeline.
    c := gen(2, 3)
    out := sq(c)


    // Consume the output.
    fmt.Println(<-out) // 4
    fmt.Println(<-out) // 9
}
由于sq的入站和出站通道具有相同的类型，我们可以将其组合任意多次。我们还可以将主函数重写为一个范围循环，就像其他阶段一样：
func main() {
    // Set up the pipeline and consume the output.
    for n := range sq(sq(gen(2, 3))) {
        fmt.Println(n) // 16 then 81
    }
}
扇出，扇入
多个函数可以从同一通道读取，直到该通道关闭；这被称为扇出。这提供了一种在一组工作者之间分配工作，以并行化CPU使用和I/O的方法。

一个函数可以从多个输入读取并在所有输入都关闭时继续进行，方法是将输入通道多路复用到一个单独的通道上，当所有输入关闭时，这个通道也会关闭。这被称为扇入。

我们可以改变我们的流水线，使其运行两个sq的实例，每个实例从同一个输入通道读取。我们引入一个新函数merge来汇总结果：
func main() {
    in := gen(2, 3)


    // Distribute the sq work across two goroutines that both read from in.
    c1 := sq(in)
    c2 := sq(in)


    // Consume the merged output from c1 and c2.
    for n := range merge(c1, c2) {
        fmt.Println(n) // 4 then 9, or 9 then 4
    }
}
merge函数通过为每个入站通道启动一个goroutine，将值复制到唯一的出站通道，从而将一组通道转换为单个通道。一旦所有的输出goroutine都已经启动，merge会再启动一个goroutine，在该通道上的所有发送都完成后关闭出站通道。

对已关闭的通道进行发送会导致恐慌，因此在调用close之前确保所有的发送都已完成非常重要。sync.WaitGroup类型提供了一种简单的方式来安排这种同步：
func merge(cs ...<-chan int) <-chan int {
    var wg sync.WaitGroup
    out := make(chan int)

    // Start an output goroutine for each input channel in cs.  output
    // copies values from c to out until c is closed, then calls wg.Done.
    output := func(c <-chan int) {
        for n := range c {
            out <- n
        }
        wg.Done()
    }
    wg.Add(len(cs))
    for _, c := range cs {
        go output(c)
    }

    // Start a goroutine to close out once all the output goroutines are
    // done.  This must start after the wg.Add call.
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}
提前终止

我们的流水线函数有一个模式：

- 阶段在所有发送操作完成后关闭出站通道。
- 阶段从入站通道接收值，直到这些通道关闭。

这个模式允许每个接收阶段被写成一个范围循环，并确保所有的goroutine在所有值成功发送到下游后退出。

但在实际的流水线中，阶段并不总是接收所有的入站值。有时这是有意的：接收者可能只需要一个值的子集来取得进展。更常见的情况是，阶段提前退出，因为入站值代表了前面阶段的错误。无论哪种情况，接收者不应该等待剩余的值到达，我们希望前面的阶段停止生成后面阶段不需要的值。

在我们的示例流水线中，如果一个阶段未能消耗所有的入站值，尝试发送这些值的goroutine将会无限期地阻塞：

```go
// 从输出中消耗第一个值。
out := merge(c1, c2)
fmt.Println(<-out) // 4 或 9
return
// 由于我们没有从out接收第二个值，
// 一个输出goroutine会陷入尝试发送它的状态。
```

这是资源泄漏：goroutine占用内存和运行时资源，而goroutine栈中的堆引用会阻止数据被垃圾回收。Goroutine不会被垃圾回收，它们必须自行退出。

我们需要在流水线的上游阶段设置，在下游阶段未能接收所有入站值时也能退出。一种方法是将出站通道更改为带有缓冲区。缓冲区可以容纳固定数量的值；如果缓冲区中有空间，则发送操作会立即完成：

```go
c := make(chan int, 2) // 缓冲区大小为 2
c <- 1  // 立即成功
c <- 2  // 立即成功
c <- 3  // 阻塞，直到另一个goroutine执行 <-c 并接收到 1
```

当在通道创建时已知要发送的值的数量时，缓冲区可以简化代码。例如，我们可以重写gen，将整数列表复制到带有缓冲区的通道中，避免创建新的goroutine：

```go
func gen(nums ...int) <-chan int {
    out := make(chan int, len(nums))
    for _, n := range nums {
        out <- n
    }
    close(out)
    return out
}
```

回到我们流水线中被阻塞的goroutine，我们可以考虑为merge返回的出站通道添加一个缓冲区：

```go
func merge(cs ...<-chan int) <-chan int {
    var wg sync.WaitGroup
    out := make(chan int, 1) // 有足够的空间存储未读取的输入
    // ... 其余部分不变 ...
```

虽然这样可以修复程序中的阻塞goroutine，但这是糟糕的代码。这里的缓冲区大小选择1取决于知道merge将接收的值的数量，以及下游阶段将消耗的值的数量。这是脆弱的：如果我们将额外的值传递给gen，或者下游阶段读取的值更少，我们将再次有阻塞的goroutine。

相反，我们需要提供一种方法，让下游阶段向发送者表明他们将停止接受输入。
显式取消

当主函数决定在未接收完所有值的情况下退出时，它必须通过在名为done的通道上发送值来告诉上游阶段的goroutine放弃它们正在尝试发送的值。它发送两个值，因为有可能有两个被阻塞的发送者：

```go
func main() {
    in := gen(2, 3)

    // 将 sq 的工作分布到两个从 in 读取的 goroutine 中。
    c1 := sq(in)
    c2 := sq(in)

    // 从输出中消耗第一个值。
    done := make(chan struct{}, 2)
    out := merge(done, c1, c2)
    fmt.Println(<-out) // 4 或 9

    // 告诉剩余的发送者我们要离开。
    done <- struct{}{}
    done <- struct{}{}
}
```

发送的goroutine使用select语句替换了其发送操作，当在out上发送发生或者从done接收到一个值时，它将继续进行。done的值类型是空结构体，因为值本身并不重要：是接收事件表示应该放弃对out的发送。输出goroutine继续在其入站通道c上进行循环，因此上游阶段不会被阻塞。（我们稍后会讨论如何允许这个循环提前返回。）

```go
func merge(done <-chan struct{}, cs ...<-chan int) <-chan int {
    var wg sync.WaitGroup
    out := make(chan int)

    // 为 cs 中的每个输入通道启动一个输出 goroutine。output
    // 从 c 复制值到 out，直到 c 关闭或者从 done 接收到一个值，然后 output 调用 wg.Done。
    output := func(c <-chan int) {
        for n := range c {
            select {
            case out <- n:
            case <-done:
            }
        }
        wg.Done()
    }
    // ... 其余部分不变 ...
```

这种方法存在一个问题：每个下游接收者都需要知道可能被阻塞的上游发送者的数量，并安排在提前返回时向这些发送者发出信号。跟踪这些计数是繁琐且容易出错的。

我们需要一种方法，可以告诉未知且未限定数量的goroutine停止向下游发送它们的值。在Go中，我们可以通过关闭通道来实现这一点，因为在关闭通道上进行接收操作总是可以立即进行，从而产生元素类型的零值。

这意味着主函数只需通过关闭done通道来解锁所有的发送者。这个关闭实际上是对发送者的广播信号。我们将每个流水线函数扩展为接受done作为参数，并通过defer语句安排关闭，以便从主函数的所有返回路径都将向流水线阶段发出信号以退出。

```go
func main() {
    // 设置一个 done 通道，它由整个流水线共享，
    // 并在流水线退出时关闭该通道，作为启动的所有 goroutine 退出的信号。
    done := make(chan struct{})
    defer close(done)          

    in := gen(done, 2, 3)

    // 将 sq 的工作分布到两个从 in 读取的 goroutine 中。
    c1 := sq(done, in)
    c2 := sq(done, in)

    // 从输出中消耗第一个值。
    out := merge(done, c1, c2)
    fmt.Println(<-out) // 4 或 9

    // done 将会在延迟调用中关闭。
}
```

现在，我们的每个流水线阶段在done被关闭时都可以自由返回。merge中的输出例程可以在不耗尽其入站通道的情况下返回，因为它知道上游的发送者sq将在done被关闭时停止尝试发送。output通过defer语句确保在所有返回路径上都调用了wg.Done：

```go
func merge(done <-chan

struct{}, cs ...<-chan int) <-chan int {
    var wg sync.WaitGroup
    out := make(chan int)

    // 为 cs 中的每个输入通道启动一个输出 goroutine。output
    // 从 c 复制值到 out，直到 c 或 done 关闭，然后调用 wg.Done。
    output := func(c <-chan int) {
        defer wg.Done()
        for n := range c {
            select {
            case out <- n:
            case <-done:
                return
            }
        }
    }
    // ... 其余部分不变 ...
```

同样地，sq可以在done被关闭时立即返回。sq确保在所有返回路径上关闭其出站通道，通过一个defer语句：

```go
func sq(done <-chan struct{}, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            select {
            case out <- n * n:
            case <-done:
                return
            }
        }
    }()
    return out
}
```

以下是流水线构建的准则：

- 阶段在所有发送操作完成后关闭出站通道。
- 阶段从入站通道接收值，直到这些通道关闭或发送者被解锁。
- 流水线通过确保所有发送的值都有足够的缓冲区，或者通过在接收者可能放弃通道时显式地向发送者发出信号来解锁发送者。

对树进行分析

让我们考虑一个更现实的流水线。

MD5是一种消息摘要算法，用作文件校验和。命令行实用程序md5sum打印一组文件的摘要值。

% md5sum *.go
d47c2bbc28298ca9befdfbc5d3aa4e65  bounded.go
ee869afd31f83cbb2d10ee81b2b831dc  parallel.go
b88175e65fdcbc01ac08aaf1fd9b5e96  serial.go

我们的示例程序类似于md5sum，但它接受一个单独的目录作为参数，并打印该目录下每个常规文件的摘要值，按路径名称排序。

% go run serial.go .
d47c2bbc28298ca9befdfbc5d3aa4e65  bounded.go
ee869afd31f83cbb2d10ee81b2b831dc  parallel.go
b88175e65fdcbc01ac08aaf1fd9b5e96  serial.go

程序的主函数调用一个名为MD5All的辅助函数，该函数返回从路径名称到摘要值的映射，然后对结果进行排序和打印：

```go
func main() {
    // 计算指定目录下所有文件的 MD5 摘要，
    // 然后按路径名称排序并打印结果。
    m, err := MD5All(os.Args[1])
    if err != nil {
        fmt.Println(err)
        return
    }
    var paths []string
    for path := range m {
        paths = append(paths, path)
    }
    sort.Strings(paths)
    for _, path := range paths {
        fmt.Printf("%x  %s\n", m[path], path)
    }
}
```

MD5All函数是我们讨论的焦点。在serial.go中，该实现没有使用并发，只是在遍历树时读取和计算每个文件的摘要。

```go
// MD5All 读取以 root 为根的文件树中的所有文件，并返回从文件路径到文件内容的 MD5 摘要的映射。
// 如果目录遍历失败或任何读取操作失败，MD5All 将返回一个错误。
func MD5All(root string) (map[string][md5.Size]byte, error) {
    m := make(map[string][md5.Size]byte)
    err := filepath.Walk(root, func(path string, info os.FileInfo, err error) error {
        if err != nil {
            return err
        }
        if !info.Mode().IsRegular() {
            return nil
        }
        data, err := ioutil.ReadFile(path)
        if err != nil {
            return err
        }
        m[path] = md5.Sum(data)
        return nil
    })
    if err != nil {
        return nil, err
    }
    return m, nil
}
```
并行摘要

在 parallel.go 中，我们将 MD5All 分割为一个两阶段的流水线。第一阶段 sumFiles 遍历树，在新的 goroutine 中对每个文件进行摘要计算，并将结果发送到一个值类型为 result 的通道中：

```go
type result struct {
    path string
    sum  [md5.Size]byte
    err  error
}
```

sumFiles 返回两个通道：一个用于结果，另一个用于 filepath.Walk 返回的错误。walk 函数为每个常规文件启动一个新的 goroutine 进行处理，然后检查 done。如果 done 被关闭，walk 会立即停止：

```go
func sumFiles(done <-chan struct{}, root string) (<-chan result, <-chan error) {
    c := make(chan result)
    errc := make(chan error, 1)
    go func() {
        var wg sync.WaitGroup
        err := filepath.Walk(root, func(path string, info os.FileInfo, err error) error {
            if err != nil {
                return err
            }
            if !info.Mode().IsRegular() {
                return nil
            }
            wg.Add(1)
            go func() {
                data, err := ioutil.ReadFile(path)
                select {
                case c <- result{path, md5.Sum(data), err}:
                case <-done:
                }
                wg.Done()
            }()
            select {
            case <-done:
                return errors.New("walk canceled")
            default:
                return nil
            }
        })
        go func() {
            wg.Wait()
            close(c)
        }()
        errc <- err
    }()
    return c, errc
}
```

MD5All 从 c 接收摘要值。MD5All 在出现错误时提前返回，并通过 defer 关闭 done：

```go
func MD5All(root string) (map[string][md5.Size]byte, error) {
    done := make(chan struct{})
    defer close(done)

    c, errc := sumFiles(done, root)

    m := make(map[string][md5.Size]byte)
    for r := range c {
        if r.err != nil {
            return nil, r.err
        }
        m[r.path] = r.sum
    }
    if err := <-errc; err != nil {
        return nil, err
    }
    return m, nil
}
```

有界并行性

parallel.go 中的 MD5All 实现为每个文件启动一个新的 goroutine。在具有许多大文件的目录中，这可能会分配比机器上可用的内存更多的内存。

我们可以通过限制并行读取的文件数量来限制这些分配。在 bounded.go 中，我们通过为读取文件创建固定数量的 goroutine 来实现这一点。我们的流水线现在有三个阶段：遍历树，读取和摘要文件，以及收集摘要。

第一阶段 walkFiles 发出树中常规文件的路径：

```go
func walkFiles(done <-chan struct{}, root string) (<-chan string, <-chan error) {
    paths := make(chan string)
    errc := make(chan error, 1)
    go func() {
        defer close(paths)
        errc <- filepath.Walk(root, func(path string, info os.FileInfo, err error) error {
            if err != nil {
                return err
            }
            if !info.Mode().IsRegular() {
                return nil
            }
            select {
            case paths <- path:
            case <-done:
                return errors.New("walk canceled")
            }
            return nil
        })
    }()
    return paths, errc
}
```

中间阶段启动了一定数量的摘要计算 goroutine，它们从 paths 接收文件名，并在通道 c 上发送结果：

```go
func digester(done <-chan struct{}, paths <-chan string, c chan<- result) {
    for path := range paths {
        data, err := ioutil.ReadFile(path)
        select {
        case c <- result{path, md5.Sum(data), err}:
        case <-done:
            return
        }
    }
}
```

与之前的示例不同，digester 不关闭其输出通道，因为多个 goroutine 在共享通道上发送。相反，在 MD5All 中的代码会安排在所有的 digesters 完成后关闭通道：

```go
c := make(chan result)
var wg sync.WaitGroup
const numDigesters = 20
wg.Add(numDigesters)
for i := 0; i < numDigesters; i++ {
    go func() {
        digester(done, paths, c)
        wg.Done()
    }()
}
go func() {
    wg.Wait()
    close(c)
}()
```

我们也可以让每个 digester 创建并返回自己的输出通道，但这样我们就需要额外的 goroutine 来将结果汇集起来。

最后一个阶段从 c 接收所有结果，然后检查 errc 的错误。此检查不能在更早的时候发生，因为在此之

前，walkFiles 可能会阻止向下发送值：

```go
m := make(map[string][md5.Size]byte)
for r := range c {
    if r.err != nil {
        return nil, r.err
    }
    m[r.path] = r.sum
}
if err := <-errc; err != nil {
    return nil, err
}
return m, nil
```

结论

本文介绍了在 Go 中构建流数据流水线的技术。处理这种流水线中的失败是棘手的，因为流水线中的每个阶段都可能阻塞，试图向下发送值，而下游阶段可能不再关心传入的数据。我们展示了如何通过关闭通道来向流水线中启动的所有 goroutine 广播“完成”信号，并定义了构建流水线的正确准则。
