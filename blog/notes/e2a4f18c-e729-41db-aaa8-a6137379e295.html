interface嵌入到struct中
笔记本：Go语言
在Go语言中，接口嵌入到结构体是一种有时令人困惑的功能。一开始可能不太清楚在结构体中嵌入接口意味着什么。本文将逐步介绍这一技巧，并给出几个实际的例子。读完本文后，你会发现其底层机制其实非常简单，而且这一技巧在多种场景中都很有用。
我们从一个简单的合成示例开始：
go
深色版本
1type Fooer interface {
2  Foo() string
3}
4
5type Container struct {
6  Fooer
7}
Fooer 是一个接口，而 Container 嵌入了它。回想第一部分的内容，当一个结构体嵌入另一个结构体时，嵌入的结构体的方法会被提升到外部结构体。对于嵌入的接口来说，情况类似；我们可以想象 Container 有一个转发方法，如下所示：
go
深色版本
1func (cont Container) Foo() string {
2  return cont.Fooer.Foo()
3}
那么 cont.Fooer 指的是什么呢？它只是实现了 Fooer 接口的任何对象。这个对象从哪里来呢？当初始化 Container 或之后，会将它赋值给 Container 中的 Fooer 字段。这里有一个例子：
go
深色版本
1// sink 接受实现了 Fooer 接口的值。
2func sink(f Fooer) {
3  fmt.Println("sink:", f.Foo())
4}
5
6// TheRealFoo 是实现了 Fooer 接口的类型。
7type TheRealFoo struct{}
8
9func (trf TheRealFoo) Foo() string {
10  return "TheRealFoo Foo"
11}
现在我们可以这样做：
go
深色版本
1co := Container{Fooer: TheRealFoo{}}
2sink(co)
这将打印出 sink: TheRealFoo Foo。
发生了什么？注意 Container 的初始化方式；嵌入的 Fooer 字段被赋值了一个 TheRealFoo 类型的值。我们只能将实现了 Fooer 接口的值赋给这个字段——其他任何值都会被编译器拒绝。由于 Fooer 接口被嵌入到了 Container 中，它的方法被提升为 Container 的方法，使得 Container 也实现了 Fooer 接口！这就是为什么我们可以将 Container 传递给 sink；如果没有嵌入，sink(co) 将无法编译，因为 co 不会实现 Fooer 接口。
你可能会好奇，如果 Container 中嵌入的 Fooer 字段未初始化会发生什么？这是一个好问题！实际上，就像你预期的那样，该字段保持默认值，对于接口来说，默认值是 nil。因此这段代码：
go
深色版本
1co := Container{}
2sink(co)
会导致运行时错误：无效的内存地址或 nil 指针引用。
这基本上涵盖了如何在结构体中嵌入接口的工作原理。接下来更重要的问题是：我们为什么需要这样做？以下是一些标准库中的用例，但我首先想从其他地方引入一个例子，展示在我看来这项技术最重要的用途之一：接口包装器。

示例：接口包装器

这个例子来源于GitHub用户 valyala 的贡献，出自他的一条评论。
假设我们需要一个带有额外功能的套接字连接，比如记录总的读取字节数。我们可以定义这样的结构体：
go
深色版本
1type StatsConn struct {
2  net.Conn
3
4  BytesRead uint64
5}
现在 StatsConn 实现了 net.Conn 接口，并且可以在期望 net.Conn 的任何地方使用。当用实现了 net.Conn 接口的值初始化 StatsConn 时，它“继承”了该值的所有方法；关键在于，我们可以拦截任何我们想要的方法，而其他方法保持不变。在这个例子中，我们想要拦截 Read 方法并记录读取的字节数：
go
深色版本
1func (sc *StatsConn) Read(p []byte) (int, error) {
2  n, err := sc.Conn.Read(p)
3  sc.BytesRead += uint64(n)
4  return n, err
5}
对于 StatsConn 的使用者来说，这种改变是透明的；我们仍然可以调用 Read 方法，并得到期望的结果（因为委托给了 sc.Conn.Read），但它还会做额外的日志记录。
如前文所述，正确初始化 StatsConn 很关键；例如：
go
深色版本
1conn, err := net.Dial("tcp", u.Host+":80")
2if err != nil {
3  log.Fatal(err)
4}
5sconn := &StatsConn{conn, 0}
这里 net.Dial 返回了一个实现了 net.Conn 的值，所以我们可以用它来初始化 StatsConn 的嵌入字段。
我们现在可以把 sconn 传给任何期望 net.Conn 参数的函数，例如：
go
深色版本
1resp, err := ioutil.ReadAll(sconn)
2if err != nil {
3  log.Fatal(err)
4}
稍后我们可以访问它的 BytesRead 字段获取总数。
这是接口包装的一个例子。我们创建了一个新的类型，它实现了已有的接口，但是重用了嵌入的值来实现大部分功能。我们可以在不使用嵌入的情况下实现这一点，例如使用显式的 conn 字段：
go
深色版本
1type StatsConn struct {
2  conn net.Conn
3
4  BytesRead uint64
5}
然后为 net.Conn 接口中每一个方法写转发方法，例如：
go
深色版本
1func (sc *StatsConn) Close() error {
2  return sc.conn.Close()
3}
然而，net.Conn 接口有8个方法。为所有这些方法写转发方法既繁琐又没有必要。嵌入接口给我们免费提供了所有这些转发方法，而我们只需覆盖我们真正需要的方法。

示例：sort.Reverse

Go 标准库中使用接口嵌入到结构体的经典例子是 sort.Reverse。对于Go新手而言，这个函数的用法常常让人困惑，因为它的工作方式并不直观。
让我们从一个简单的排序例子开始，对整数切片进行排序。
go
深色版本
1lst := []int{4, 5, 2, 8, 1, 9, 3}
2sort.Sort(sort.IntSlice(lst))
3fmt.Println(lst)
这将输出 [1 2 3 4 5 8 9]。它是怎么工作的？sort.Sort 函数接受实现了 sort.Interface 接口的参数，该接口定义如下：
go
深色版本
1type Interface interface {
2    // Len 是集合中的元素数量。
3    Len() int
4    // Less 报告索引 i 的元素是否应该排在索引 j 的元素之前。
5    Less(i, j int) bool
6    // Swap 交换索引 i 和 j 的元素。
7    Swap(i, j int)
8}
如果我们有一种类型想要用 sort.Sort 排序，我们就必须实现这个接口；对于像整数切片这样简单的类型，标准库提供了像 sort.IntSlice 这样的便利类型，它们接收我们的值并为该值实现了 sort.Interface 方法。到目前为止一切顺利。
那么 sort.Reverse 是怎么工作的呢？通过巧妙地利用嵌入到结构体中的接口。sort 包中有一个（未导出的）类型帮助完成任务：
go
深色版本
1type reverse struct {
2  sort.Interface
3}
4
5func (r reverse) Less(i, j int) bool {
6  return r.Interface.Less(j, i)
7}
此时应该很清楚这是做什么的了；reverse 通过嵌入它（只要用实现了接口的值初始化）来实现 sort.Interface，并且它拦截了接口中的一个方法 Less。然后它将 Less 委托给嵌入的值，但是反转了参数顺序。这个 Less 实际上比较了元素的逆序，这会让排序结果反转。
为了完成解决方案，sort.Reverse 函数很简单：
go
深色版本
1func Reverse(data sort.Interface) sort.Interface {
2  return &reverse{data}
3}
现在我们可以这样做：
go
深色版本
1sort.Sort(sort.Reverse(sort.IntSlice(lst)))
2fmt.Println(lst)
这将输出 [9 8 5 4 3 2 1]。理解的关键在于调用 sort.Reverse 本身不会排序或反转任何东西。它可以看作是一个高阶函数：它产生一个值，这个值包装了传入的接口并调整其功能。调用 sort.Sort 才是真正进行排序的地方。

示例：context.WithValue

context 包有一个叫做 WithValue 的函数：
go
深色版本
1func WithValue(parent Context, key, val interface{}) Context
它“返回一个 parent 的副本，其中与 key 关联的值是 val”。我们来看看它内部是如何工作的。
忽略错误检查，WithValue 大致可以简化为：
go
深色版本
1func WithValue(parent Context, key, val interface{}) Context {
2  return &valueCtx{parent, key, val}
3}
其中 valueCtx 是：
go
深色版本
1type valueCtx struct {
2  Context
3  key, val interface{}
4}
这里又是结构体嵌入接口的情况。valueCtx 现在实现了 Context 接口，并且可以自由地拦截 Context 的4个方法中的任何一个。它拦截了 Value 方法：
go
深色版本
1func (c *valueCtx) Value(key interface{}) interface{} {
2  if c.key == key {
3    return c.val
4  }
5  return c.Context.Value(key)
6}
而其他方法保持不变。

示例：降级能力，使用更受限的接口

这个技巧相当高级，但在标准库的很多地方都有使用。不过，我认为对于客户端代码来说通常不需要用到这个技巧，所以如果你是Go新手，第一次阅读时没有完全理解也没关系，随着你的Go经验增长，再回过头来看这个技巧。
我们先来谈谈 io.ReaderFrom 接口：
go
深色版本
1type ReaderFrom interface {
2    ReadFrom(r Reader) (n int64, err error)
3}
这个接口被那些可以从 io.Reader 有意义地读取数据的类型实现。例如，os.File 类型实现了这个接口，并从读取器读取数据到它代表的打开的文件中。我们来看看它是如何做到这一点的：
go
深色版本
1func (f *File) ReadFrom(r io.Reader) (n int64, err error) {
2  if err := f.checkValid("write"); err != nil {
3    return 0, err
4  }
5  n, handled, e := f.readFrom(r)
6  if !handled {
7    return genericReadFrom(f, r)
8  }
9  return n, f.wrapErr("write", e)
10}
它首先尝试使用 readFrom 方法从 r 读取数据，该方法是操作系统特定的。例如，在Linux上，它使用 copy_file_range 系统调用来在内核中直接快速复制两个文件之间的数据。
readFrom 返回一个布尔值表示是否成功（handled）。如果不成功，ReadFrom 会尝试使用 genericReadFrom 进行“通用”操作，其实现如下：
go
深色版本
1func genericReadFrom(f *File, r io.Reader) (int64, error) {
2  return io.Copy(onlyWriter{f}, r)
3}
它使用 io.Copy 从 r 复制到 f，到目前为止一切顺利。但是 onlyWriter 包装器是什么呢？
go
深色版本
1type onlyWriter struct {
2  io.Writer
3}
很有趣。这是我们现在已经熟悉的——结构体中嵌入接口的技巧。但如果我们在文件中查找，我们会发现 onlyWriter 上并没有定义任何方法，也就是说它没有拦截任何方法。那么为什么需要它呢？
要理解为什么需要 onlyWriter，我们应该看看 io.Copy 是怎么做的。它的代码很长，我在这里不会完整地复现它；但需要注意的关键部分是，如果 io.Copy 的目标实现了 io.ReaderFrom 接口，它将会调用 ReadFrom 方法。但这又回到了起点，因为我们是在调用 File.ReadFrom 时进入 io.Copy 的。这会导致无限递归！
现在开始明白为什么需要 onlyWriter 了。通过在调用 io.Copy 时用 onlyWriter 包装 f，io.Copy 接收到的不是一个实现了 io.ReaderFrom 接口的类型，而只是一个实现了 io.Writer 接口的类型。然后它会调用我们的 File 的 Write 方法，从而避免了 ReadFrom 导致的无限递归陷阱。
正如我之前提到的，这个技巧属于较为高级的范畴。我觉得重要的是要突出这一点，因为它代表了“在结构体中嵌入接口”工具的明显不同的使用方式，并且在标准库中广泛使用。
在 File 中的使用是一个好的例子，因为它给 onlyWriter 提供了一个明确命名的类型，有助于理解它的作用。标准库中的一些代码避开了这种自我文档化的模式，而是使用匿名结构体。例如，在 tar 包中是这样做的：
go
深色版本
1io.Copy(struct{ io.Writer }{sw}, r)
通过这种方式，它创建了一个临时的匿名结构体，只包含 io.Writer 接口，从而确保 io.Copy 只调用 Write 方法而不会尝试调用 ReadFrom 方法。这种做法减少了无限递归的风险，并保持了代码简洁。虽然这种方法不如给类型命名那样直观易懂，但它仍然是一个有效的使用接口嵌入的案例。
